Database Fundamentals

1. WHAT ARE DATABASES

A Database is a shared collection of logically related data and a description of these data, designed to meet the information needs of an organization

  • Data Storage: A database is used to store large amounts of structured data, making it easily accessible, searchable, and retrievable.
  • Data Analysis: A database can be used to perform complex data analysis, generate reports, and provide insights into the data.
  • Record Keeping: A database is often used to keep track of important records, such as financial transactions, customer information, and inventory levels.
  • Web Applications: Databases are an essential component of many web applications, providing dynamic content and user management.

C.R.U.D.E in SQL

It is an extension of the CRUD operations, which stand for:

  • C - Create → INSERT INTO (Add new records)
  • R - Read → SELECT (Retrieve data)
  • U - Update → UPDATE (Modify existing records)
  • D - Delete → DELETE (Remove records)
  • Execute → Running queries, stored procedures, or transactions.
  • Export → Extracting data from the database (e.g., SELECT INTO, exporting CSV files, etc.)

2. PROPERTIES OF AN IDEAL DATABASE

  • Integrity: Ensures accuracy, consistency, and reliability of data.
    • Enforced using constraints (e.g., PRIMARY KEY, FOREIGN KEY, CHECK).
    • Supports ACID properties (Atomicity, Consistency, Isolation, Durability).
  • Availability: Ensures data is accessible whenever needed, even during failures.
    • Achieved through replication, clustering, and failover mechanisms.
    • High availability databases minimize downtime.
  • Security: Protects data from unauthorized access, breaches, and corruption.
    • Implements authentication (user roles, passwords) and authorization (permissions, access controls).
    • Uses encryption, firewalls, and auditing mechanisms.
  • Independent of Application:
    • The database should be decoupled from specific applications.
    • Supports multiple platforms and programming languages.
    • Allows changes in the database structure without affecting applications (schema evolution).
  • Concurrency: Allows multiple users to access and modify data simultaneously.
    • Uses locking mechanisms (pessimistic & optimistic locking) to prevent conflicts.
    • Ensures data consistency through transaction management and isolation levels.

3. TYPES OF DATABASES

  • Relational Databases: Use a relational model to organize data into tables with rows and columns.
  • NoSQL Databases: Designed to handle large amounts of unstructured or semi-structured data. (MongoDB)
  • Column Databases: Store data in columns rather than rows. (Amazon Redshift, Google Big Query)
  • Graph Databases: Store and query graph-structured data. (Neo4j, Amazon Neptune)
  • Key-value Databases: Store data as key-value pairs. (Redis, Amazon DynamoDB)

4. WHAT IS A DBMS

A database management system (DBMS) is a software system that provides the interfaces and tools needed to store, organize, and manage data in a database. A DBMS acts as an intermediary between the database and the applications or users that access the data stored in the database.

6. DATABASE KEYS

A key in a database is an attribute or a set of attributes that uniquely identifies a tuple (row) in a table. Keys play a crucial role in ensuring the integrity and reliability of a database by enforcing unique constraints on the data and establishing relationships between tables.

  • Super Key: A Super key is a combination of columns that uniquely identifies any row within a relational database management system (RDBMS) table
  • Candidate Key: A candidate key is a minimal Super key, meaning it has no redundant attributes.
  • Primary Key: A primary key is a unique identifier for each tuple in a table. It cannot contain null values.
  • Alternate Key: A candidate key that is not used as the primary key.
  • Composite Key: A primary key made up of two or more attributes.

7. CARDINALITY OF RELATIONSHIPS

Cardinality in database relationships refers to the number of occurrences of an entity in a relationship with another entity. Cardinality defines the number of instances of one entity that can be associated with a single instance of the related entity.

8. DRAWBACKS OF DATABASES

  • Complexity: Setting up and maintaining a database is complex and time-consuming
  • Cost: High expenses are involved in database setup and maintenance, including hardware, software
  • Scalability: Managing large and growing databases can lead to performance and scalability.
  • Data Integrity: Ensuring accurate and consistent data is difficult, especially with multiple users updating it simultaneously.
  • Security: Protecting databases from unauthorized access and cyber threats is a major challenge.
  • Data Migration: Transferring or upgrading databases is often a complex and time-consuming process.
  • Flexibility: Rigid database structures make adapting to new requirements or data types difficult.
